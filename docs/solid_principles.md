# SOLID Принципы в Приложении

Этот документ описывает применение SOLID принципов в разработке приложения.

## Single Responsibility Principle (SRP)
Каждый класс имеет одну причину для изменения:
- `Ingredient` - представляет один ингредиент с его питательной ценностью
- `Dish` - представляет одно блюдо
- `IngredientDataLoader` - отвечает только за загрузку данных об ингредиентах
- `DishLoader` - отвечает только за загрузку данных о блюдах
- `NutritionCalculator` - отвечает только за расчет питательной ценности

## Open/Closed Principle (OCP)
Код открыт для расширения, но закрыт для модификации:
- Интерфейсы в `src/models/interfaces.py` позволяют легко добавлять новые реализации
- Например, можно создать `DatabaseIngredientLoader`, реализующий `IngredientLoaderInterface`

## Liskov Substitution Principle (LSP)
Объекты могут быть заменены их наследниками без изменения корректности программы:
- Все классы, реализующие интерфейсы, могут быть использованы вместо базовых классов

## Interface Segregation Principle (ISP)
Многочисленные специализированные интерфейсы лучше, чем один общий:
- Отдельные интерфейсы для разных типов компонентов (загрузчики, калькуляторы)
- Каждый компонент реализует только те интерфейсы, которые ему нужны

## Dependency Inversion Principle (DIP)
Зависимости должны быть от абстракций, а не от конкретных реализаций:
- В `app.py` используются интерфейсы вместо конкретных классов
- Компоненты зависят от `IngredientLoaderInterface`, а не от `IngredientDataLoader` напрямую
